## Published Image

The server image is published to GHCR as:

```text
ghcr.io/vaylix/vaylix:latest
```

Pull it directly:

```sh
docker pull ghcr.io/vaylix/vaylix:latest
```

## Default Data Directory

The container and server default to:

```text
/var/lib/vaylix
```

Use that as the mounted volume path:

```sh
docker run \
  -p 9173:9173 \
  -v vaylix-data:/var/lib/vaylix \
  -e VAYLIX_USER=vaylix \
  -e VAYLIX_PASSWORD=vaylix \
  ghcr.io/vaylix/vaylix:latest
```

Auth and outbound compression are enabled by default. TLS still needs explicit certificate configuration if the container is exposed outside a trusted local network. mTLS additionally requires mounting the client CA bundle and passing `--tls-client-ca`.

## Supported Environment Variables

These environment variables are read by the server binary inside the container:

| Variable | Purpose | Default in image |
| --- | --- | --- |
| `VAYLIX_BIND` | Bind address for the server listener | `0.0.0.0` |
| `VAYLIX_PORT` | TCP port for the server listener | `9173` |
| `VAYLIX_MAX_CONNECTIONS` | Maximum concurrent sessions | `256` |
| `VAYLIX_SSL` | Enable TLS transport | `false` |
| `VAYLIX_TLS_CERT` | Path to PEM certificate chain | unset |
| `VAYLIX_TLS_KEY` | Path to PEM private key | unset |
| `VAYLIX_TLS_CLIENT_CA` | Path to PEM CA bundle for mTLS client verification | unset |
| `VAYLIX_DATA_DIR` | Persistence root mounted into the container | `/var/lib/vaylix` |
| `VAYLIX_WAL_SYNC` | WAL durability mode: `buffered`, `flush`, or `sync` | `flush` |
| `VAYLIX_USER` | Required username when auth is enabled | `vaylix` |
| `VAYLIX_PASSWORD` | Required password when auth is enabled | `vaylix` |
| `VAYLIX_DISABLE_AUTH` | Disable auth for trusted local/test use | `false` |
| `VAYLIX_DISABLE_COMPRESSION` | Disable outbound frame compression | `false` |

Other server limits and maintenance intervals currently remain CLI flags rather than environment
variables.

## TLS and mTLS Example

If you need TLS or mTLS in the container, mount the certificate material and set the matching
runtime variables:

```sh
docker run \
  -p 9173:9173 \
  -v vaylix-data:/var/lib/vaylix \
  -v "$(pwd)/certs:/certs:ro" \
  -e VAYLIX_SSL=true \
  -e VAYLIX_TLS_CERT=/certs/server.crt \
  -e VAYLIX_TLS_KEY=/certs/server.key \
  -e VAYLIX_TLS_CLIENT_CA=/certs/ca.crt \
  -e VAYLIX_USER=vaylix \
  -e VAYLIX_PASSWORD=vaylix \
  ghcr.io/vaylix/vaylix:latest
```

## What Lives in the Data Directory

- `snapshot.bin`
- `snapshot.bin.tmp`
- `manifest.bin`
- `manifest.bin.tmp`
- `wal.log`
- `keyring.bin`
- `keyring.bin.tmp`
- `audit.log`

## Why the Volume Matters

Without a mounted volume, you lose:

- snapshots
- WAL history
- the server-managed storage keyring
- audit logs

That means a fresh ephemeral container would not be able to recover old encrypted data even if you copied only part of the directory. The data directory is the persistence boundary, not just the snapshot file.

Do not mount only `snapshot.bin` or only `wal.log`. The keyring and manifest are required to verify and decrypt persisted data.
