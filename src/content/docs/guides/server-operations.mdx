## Important Server Flags

| Flag | Purpose |
| --- | --- |
| `--bind` | Listener address |
| `--port` | Listener port |
| `--data-dir` | Persistent storage root |
| `--user` / `--password` | Auth credentials; defaults are local-development values |
| `--disable-auth` | Disable authentication for trusted local testing only |
| `--ssl` | Enable TLS |
| `--tls-cert` / `--tls-key` | TLS certificate and private key |
| `--tls-client-ca` | Require client certificates signed by this CA bundle |
| `--wal-sync` | WAL durability policy |
| `--snapshot-interval-seconds` | Periodic background snapshotting |
| `--expiration-sweep-interval-seconds` | Background TTL sweeper |
| `--idle-timeout-seconds` | Disconnect idle clients |
| `--disable-compression` | Disable default outbound zstd frame compression |
| `--audit-log-path` | Override audit log location |

## Default Runtime Behavior

Unless configured otherwise, the server starts with:

- plaintext TCP transport
- auth enabled
- outbound zstd frame compression enabled
- server-managed encrypted persistence under `--data-dir`
- append-only JSONL audit logging

Enable TLS explicitly for production-facing deployments:

```sh
cargo run -p server -- \
  --ssl \
  --tls-cert ./certs/server.crt \
  --tls-key ./certs/server.key
```

Require mTLS by adding a client CA bundle:

```sh
cargo run -p server -- \
  --ssl \
  --tls-cert ./certs/server.crt \
  --tls-key ./certs/server.key \
  --tls-client-ca ./certs/client-ca.crt
```

When `--tls-client-ca` is set, clients must present a certificate chaining to that CA. Application auth still runs after the TLS handshake unless `--disable-auth` is explicitly used for trusted testing.

## Abuse Controls

The server currently enforces:

- request payload byte limits
- key size limits
- value size limits
- batch key-count limits
- per-session transaction queue limits
- token-bucket request rate limiting
- idle connection timeouts

These are runtime guardrails, not a full denial-of-service strategy.

## Audit Logging

Audit logging is enabled automatically. By default the server writes:

```text
<data-dir>/audit.log
```

Override it if needed:

```sh
cargo run -p server -- --audit-log-path /var/log/vaylix/audit.log
```

## WAL Durability Modes

- `buffered`: let the OS decide when to flush
- `flush`: flush userspace buffers after append
- `sync`: call through to kernel sync for each append

For most local or early-stage deployments, `flush` is the practical default.
