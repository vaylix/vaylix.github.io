## Core Stack

`client -> transport -> TCP/TLS -> transport -> server -> engine`

That line is the main architectural rule.

- REPL

## How the Pieces Connect

The repository structure mirrors the runtime layering directly:

- `client` turns user input into `command` values and sends them through `transport`
- `transport` owns framing, opcodes, checksums, and message encoding for both sides
- `server` accepts TCP or TLS connections, authenticates sessions, enforces guardrails, and routes decoded requests
- `engine` applies state changes, maintains durability, and recovers from persisted storage

That separation is the reason transport changes do not require socket parsing inside the engine, and engine changes do not require protocol logic inside storage code.

## Why the Engine Worker Exists

The current server accepts multiple client connections concurrently, but mutating engine work is still serialized through a dedicated engine worker. That gives:

- deterministic single-node ordering
- a clean place for atomic transaction application
- less shared-state complexity in the async server path

It is a deliberate single-node design, not a distributed concurrency model.
